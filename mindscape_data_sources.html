Mindscape v1 · Data Model

What data we use, and why

Mindscape does not discover the product problem from its own data. The product problem is established by external public evidence. The app then collects a narrow set of runtime signals to support early intervention at cohort level and opt-in follow-up at student level.

External evidence

Used to justify the problem

WHO, Healthy Minds, AUCCCD, UGC, Tele-MANAS, and screening literature define the campus mental-health burden and help-seeking gap.

Institution setup

Used to give context

Institution profile, cohorts, department sizes, and calendar events like exams, lab practicals, and placement weeks.

Runtime student data

Used to drive the product

Anonymous daily pulse, weekly PHQ-2/GAD-2, safety status, resource request, and explicit support opt-in.

V1 Collection

Layer Data Purpose
Daily pulse Mood 1-10, stress 1-5, energy 1-5, sleep quality 1-5, connectedness 1-5, optional 160-char reflection, safety status, wants resources now Creates a daily distress profile and trend line without clinical overload.
Weekly screener PHQ-2 + GAD-2, weekly support preference, safety status Anchors the product in a validated short screening layer.
Operational Response gaps, 7-day trend, weekly completion, repeated high-concern pulses, support-request volume Improves cohort-level triage and dashboard signal quality.
Institution context Cohort, department, exam weeks, placement weeks, lab practical periods, local resource directory Helps counseling teams interpret spikes in context.
Opt-in only Name, email, phone, preferred contact, support note Stored only when a student explicitly requests follow-up.

Do Use

  • Validated short screeners.
    PHQ-2 and GAD-2 are compact enough for weekly use and evidence-backed enough for a student product.
  • Trend over time.
    The dashboard should care about repeated strain and falling trajectory, not only one bad day.
  • Explicit support intent.
    If a student requests help, that is a stronger operational signal than any passive guess.

Use Softly

  • Response gaps.
    Useful as an operational cue, but not sufficient on their own for a “risk” claim.
  • Reflection themes.
    Good for resource routing and theme clustering; not the primary risk engine.
  • Calendar overlays.
    Interpretive context only, not a clinical signal.

Do Not Use in V1

  • No passive sensing core.
    Phone telemetry, GPS, contacts, and similar signals stay out of the v1 product.
  • No hidden identification.
    Institutions cannot reverse anonymous pulse data into a student identity.
  • No black-box diagnosis.
    Mindscape is an early-support tool, not a diagnostic system.
