Mindscape v1 · Privacy Model

Anonymous by default, identifiable only by request

Mindscape should treat its data as sensitive wellness data. The privacy model is not “this is not health data”; it is “this is wellness data handled with strict minimization and an explicit identity boundary.”

Default state

Anonymous runtime

Daily pulses and weekly screens are linked to a pseudonymous device hash only. Admins see cohort-level risk, never a student identity.

Identity boundary

Separate support queue

Name, email, or phone are captured only in the opt-in support request form and stored in a separate support request record.

Default posture

Minimize and expire

No GPS, no contacts, no raw passive sensing, no hidden cross-linking, and retention windows should be explicit in production.

Explicit exclusions

  • No GPS, contacts, call logs, messages, or browsing data.
  • No hidden student re-identification from pulse data.
  • No claim that administrators can see “individual risk” in anonymous mode.
  • No passive sensing in v1.

Production retention defaults

  • Daily pulse: retain detailed row-level records for operational analytics, then aggregate or prune on a defined retention schedule.
  • Reflection text: keep only shortened excerpts or derived tags; avoid indefinite raw-text storage.
  • Support request identity: retain only as long as needed for counseling follow-up and audit obligations.
  • Logs: never log raw contact details or support-note content.
